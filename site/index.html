INDUSTRIAL IOT · ON REAL HARDWARE

How factory data moves: MQTT and OPC UA.

A small but complete industrial-telemetry pipeline built on hardware you can hold. Sensor readings from an ESP32-S3 board are published over MQTT into a C#/.NET service that stores them, and the same readings are re-exposed through an OPC UA server that any standard industrial client can browse. It is the shape of a real manufacturing data platform, in miniature.

MQTT Mosquitto OPC UA Node-OPCUA historian C# / .NET + SQLite bridge TypeScript device ESP32-S3

01 The pipeline

One device publishes. A broker routes. Two independent consumers store and re-expose the data. None of them knows the others exist; the broker decouples everything.

Add a third consumer tomorrow and the device's code does not change at all. That decoupling is the point.

Why a broker instead of calling the device directly? Direct calls couple every consumer to the device: each one has to know its address, poll it on its own schedule, and handle its outages. With a broker in the middle, the device publishes once and any number of consumers subscribe, none of them aware of the others. That is why publish/subscribe owns the IoT world.

02 Two protocols, two shapes

MQTT and OPC UA are unrelated standards that solve two halves of the same problem. The clearest way to hold them apart is their shape.

Same goal, two shapes. MQTT: announce through a middleman. OPC UA: every machine answers for itself.

MQTT moves data
A featherweight publish/subscribe protocol: many devices stream tiny messages through a central broker to many listeners, over cheap chips and unreliable networks. It does not care what the bytes mean.
OPC UA describes data
A vendor-neutral standard where every machine runs a server that presents a browsable, typed tree of its values. One generic client can discover and read any vendor's server with no custom code.
The gateway is a choice
The two standards are strangers; neither needs the other. This project's OPC UA server just happens to listen to the MQTT side and answer OPC UA visitors. That wiring lives in the code, not in either standard.

03 The proof: live in a real client

The payoff is watching the board's own readings appear in UaExpert, the industry's free reference OPC UA client, with zero custom client code.

UaExpert connected to opc.tcp://localhost:4840, browsing Objects > MatrixBoard. The five nodes read Good, and the accelerometer values update as the physical board is moved. (The server's machine hostname is blurred.)

Browsable by design
A client that knew nothing about this server discovered every value by browsing. That discovery-without-documentation property is the whole argument for OPC UA over a pile of custom REST APIs.
Timestamps that lag on purpose
A subscription reports only when a value changes. A still sensor keeps its older timestamp honestly, rather than pretending to refresh.
A live negotiation
The client asks for a sampling interval; the server enforces its own floor and revises the client up to it. Client requests, server disposes, all visible in the log.

An honest label: ChipTemperatureC is the temperature of the ESP32's own silicon, not the room. The board has no ambient temperature sensor, so the value is named for exactly what it is.

04 Run it yourself

Four small pieces, each started on its own. Node 22.6+ runs the TypeScript directly, no build step.

# 1. broker (Docker Desktop must be running)
docker compose up -d

# 2. bridge: poll the board's HTTP API, publish to MQTT
node bridge/bridge.ts

# 3. historian: subscribe and store every reading in SQLite
dotnet run --project historian

# 4. OPC UA server: re-expose the readings as browsable nodes
node opcua/server.ts

Then verify at the destination, never the source's logs: node scripts/dbcheck.js reads row counts and timestamps straight from the database, and node scripts/opcuacheck.ts connects the way a real client would and reads live values back.
